4.1 Eucalyptus & OpenStack — Architecture, Features, Modes of Operation

Eucalyptus

Eucalyptus (Elastic Utility Computing Architecture for Linking Your Programs To Useful Systems) is an open-source platform for building private and hybrid clouds. It is compatible with Amazon EC2/S3 APIs, meaning you can use the same tools you use with AWS on a private cloud.

Architecture Components

☁ Cloud Controller (CLC)
  • Top-level component (brain of the cloud)
  • Manages entire cloud resources
  • Handles user/admin interface
  • Interacts with other components
🏢 Cluster Controller (CC)
  • Manages a group of nodes (one per availability zone)
  • Schedules VMs on node controllers
  • Manages virtual networking
💻 Node Controller (NC)
  • Runs on each physical machine
  • Directly manages VMs (using KVM/Xen hypervisor)
  • Reports resource availability to CC
📦 Walrus Storage Controller (WSC)
  • S3-compatible object storage
  • Stores VM images, snapshots, user data
🗄 Storage Controller (SC)
  • EBS-compatible block storage
  • Provides persistent storage for VMs
🌐 VMware Broker (optional)
  • Integrates with VMware infrastructure
  • Allows hybrid setups
Cloud Controller (CLC) | ───────────────── | | Cluster Walrus Controller (Object Storage) (CC) | | Storage | Controller (SC) ────────────────────── | | | NC1 NC2 NC3 (VMs) (VMs) (VMs)

Modes of Operation

MANAGED Mode
  • Full network management by Eucalyptus
  • Uses VLAN tagging for isolation
  • Supports security groups, elastic IPs
  • Most feature-rich mode
MANAGED-NOVLAN Mode
  • Like MANAGED but without VLANs
  • Simpler networking — no VLAN-capable switches needed
  • Still supports security groups
SYSTEM Mode
  • Least control — physical network handles everything
  • VMs get IPs from the existing DHCP server
  • No isolation between users
STATIC Mode
  • Static IPs assigned manually to VMs
  • Eucalyptus manages IP assignment from a pool
  • Good for stable, fixed IP environments

Key Features of Eucalyptus

  • AWS-compatible APIs (EC2, S3, EBS, IAM)
  • Supports multiple hypervisors: KVM, Xen, VMware
  • Private and hybrid cloud deployment
  • Role-based access control
  • VM image management
  • Open-source and highly customizable

OpenStack

OpenStack is a free, open-source cloud computing platform used to build and manage both public and private clouds. It provides Infrastructure as a Service (IaaS) through a set of interrelated services, each controlling a specific aspect of the cloud.

Key Components (Services)

Nova (Compute)
  • Core of OpenStack — manages VMs
  • Handles scheduling, creating, and terminating VMs
  • Works with hypervisors: KVM, Xen, VMware, Hyper-V
  • Sub-parts: Nova API, Scheduler, Conductor, Nova-compute
Neutron (Networking)
  • Manages networking for OpenStack
  • Creates virtual networks, routers, subnets
  • Supports VLANs, VxLAN, plugins
  • DHCP/IPAM and firewall services
Cinder (Block Storage)
  • Persistent block storage (like a hard drive for VMs)
  • Similar to AWS EBS
  • Manages volumes, snapshots, backups
Swift (Object Storage)
  • Stores unstructured data — files, images, backups
  • Like AWS S3
  • Proxy server + Object store architecture
  • Highly scalable and redundant
Keystone (Identity)
  • Authentication and authorization service
  • Manages users, roles, tenants, tokens
  • Central "gatekeeper" — all services talk to it
Glance (Image Service)
  • Stores and retrieves VM disk images
  • Supports multiple formats: QCOW2, VMDK, RAW
  • Glance API + Glance DB + Registry
Horizon (Dashboard)
  • Web-based UI for managing OpenStack
  • Admin and user dashboards
  • Provides a graphical alternative to CLI
Ceilometer (Telemetry)
  • Collects usage/monitoring data
  • Used for billing, metering, alerting
  • Components: Ceilometer API, Agent, Collector
User → Horizon (UI) or CLI | ─────────────────────────────────────────────────── | | | | | | Nova Neutron Cinder Swift Glance Ceilometer (Compute) (Network) (Block) (Object) (Image) (Metering) | | Compute Network Node Block Storage Node DHCP/Router (Physical disk) (VMs+Hypervisor) | All services authenticate via KEYSTONE
Easy way to remember OpenStack services: Nova (VMs), Neutron (Networks), Cinder (block Storage), Swift (object Storage), Keystone (Keys/auth), Glance (images), Horizon (HTML dashboard), Ceilometer (Counting usage).

Google App Engine (GAE)

Google App Engine is a fully managed PaaS (Platform as a Service) offered by Google Cloud Platform (GCP). You just write your app code, and Google handles everything else — servers, scaling, OS updates, etc.

4.3 IoT and Cloud Computing

What is IoT?

Internet of Things (IoT) refers to the network of physical devices (sensors, cameras, smart appliances, vehicles, etc.) that are connected to the internet and can collect and share data automatically — without human intervention.

Examples: Smart thermostats, fitness bands, smart home devices, industrial sensors, connected cars, hospital patient monitors.

IoT and Cloud Computing — Why They Need Each Other

What IoT Needs from Cloud
  • Massive storage for sensor data
  • Computing power to analyze data in real-time
  • Scalability (millions of devices)
  • Remote management of devices
  • Cost-effective infrastructure (pay-as-you-go)
What Cloud Gets from IoT
  • Continuous stream of real-world data
  • New use cases (smart cities, healthcare, industry)
  • Revenue from IoT services
  • Greater adoption across industries

IoT Working Model with Cloud

IoT Devices Edge/Gateway Cloud Platform Applications (Sensors, → (Local Processing → (AWS IoT, → (Dashboard, Cameras, & data filtering) Azure IoT Hub, Alerts, Analytics, Actuators) Google IoT Core) Mobile Apps)

Benefits of IoT Cloud (Functions of IoT Cloud)

  • Scalability — Easily scale from 100 to 1 million devices
  • Data Storage — Cloud stores massive amounts of sensor time-series data
  • Real-time Analytics — Process and analyze data as it arrives (stream processing)
  • Remote Management — Update device firmware, monitor health, reboot remotely
  • Cost Efficiency — No need for own data centers; pay for what you use
  • Security — Cloud provides encryption, authentication for devices
  • AI/ML Integration — Run machine learning on IoT data for predictions (predictive maintenance)
  • High Availability — Cloud ensures data is always accessible

Comparison: IoT vs Cloud Computing

Aspect IoT Cloud Computing
Definition Network of physical devices generating real-world data On-demand computing resources delivered over internet
Location At the edge — devices are distributed everywhere Centralized — data centers in specific locations
Data Generation Generates data (sensors, cameras) Processes and stores data
Processing Limited local processing (edge) Massive centralized processing power
Latency Real-time response needed (low latency) Some latency due to distance from devices
Relationship IoT generates data that cloud processes Cloud provides storage/compute for IoT

Role of Cloud Computing in IoT

  • Device Management: Register, authenticate, monitor, and update IoT devices at scale
  • Data Ingestion: Handle millions of concurrent data streams (AWS Kinesis, Azure Event Hub)
  • Storage: Store historical sensor data cheaply in cloud databases (time-series DBs)
  • Processing: Run analytics, ML models on IoT data
  • Notification: Trigger alerts and actions based on sensor readings

Cloud IoT Platforms

  • AWS IoT Core — connects billions of devices, integrates with Lambda, S3
  • Microsoft Azure IoT Hub — enterprise IoT, Azure Digital Twins
  • Google Cloud IoT Core — analytics with BigQuery, Pub/Sub
Key Quote for Exam: "Cloud computing enables scalability, whereas IoT enables context-awareness. The real power lies in their convergence." IoT devices collect context (temperature, location, state); cloud provides the scale to process it.

4.4 Introduction to Edge Computing & Fog Computing

The Problem with Pure Cloud for IoT

When IoT devices send ALL data to the cloud for processing, there are issues: high latency (delay in response), bandwidth consumption (huge data transfer costs), and privacy concerns (sensitive data leaving the device). Edge and Fog computing solve this.

Edge Computing

Edge Computing means processing data right at the source (or very close to it) — on the IoT device itself or a gateway nearby — instead of sending it all to a distant cloud data center.

How It Works

IoT Device (Temperature Sensor) | Edge Device (Raspberry Pi / Local Server) ← Data processed HERE (filter, analyze, act) | Only relevant/summarized data sent to Cloud | Cloud (Long-term storage, big analytics, AI training)

Features / Characteristics

  • Very low latency — decisions made in milliseconds locally
  • Reduces bandwidth usage — only important data goes to cloud
  • Works offline — can operate even without internet connection
  • More data privacy — sensitive data doesn't leave the premises
  • Physical devices do real-time processing (edge nodes)

Applications

  • Self-driving cars — split-second decisions can't wait for cloud
  • Manufacturing — detect defects on assembly line in real-time
  • Healthcare — real-time patient monitoring with local alerts
  • Smart cameras — face recognition processed on the camera itself

Fog Computing

Fog Computing is an extension of edge computing that creates a distributed computing layer between IoT devices and the cloud. Fog nodes (like routers, gateways, micro-servers) act as an intermediate processing layer. Think of it as "cloud brought closer to the ground" — like fog is a cloud that's low to the earth.

Fog Architecture (3-Layer Model)

Layer 3: Cloud Layer (Centralized storage, deep analytics, AI training) ↑ summarized data only Layer 2: Fog Layer ← THE KEY LAYER (Gateways, Fog Nodes, Mini-servers) (Filter, aggregate, process data from below) ↑ raw or filtered data Layer 1: Device Layer (IoT Sensors, Cameras, Smart Devices) (Generate data continuously)

Features of Fog Computing

  • Distributed architecture — multiple fog nodes spread geographically
  • Supports mobility — fog nodes can be mobile (vehicles, drones)
  • Low latency for time-sensitive tasks
  • Can process data from heterogeneous devices (different protocols)
  • Reduces cloud workload
  • Backed by Cisco (coined the term)

Comparison: Edge vs Fog vs Cloud

Aspect Edge Computing Fog Computing Cloud Computing
Location On or near the device Between device and cloud (gateways) Centralized data centers
Latency Lowest (milliseconds) Low (milliseconds to seconds) Higher (seconds)
Compute Power Limited (on-device) Moderate (fog nodes) Unlimited (scalable)
Storage Very limited Moderate Unlimited
Internet Required No (can be offline) Partial Yes (always)
Best Use Case Real-time, mission-critical tasks Smart city, industrial IoT Long-term analytics, training

5.1 Security Issues in Cloud — Threats, Information Security, Identity Management, Design Principles, Frameworks

Why is Cloud Security Different?

In traditional computing, all data and systems are within your company's walls. In cloud computing, your data lives on someone else's infrastructure, accessed over the internet, often shared with other users (multi-tenancy). This creates unique security challenges.

Major Cloud Security Threats

Data Breaches

Unauthorized access to sensitive cloud-stored data. Often due to misconfigured access controls.

Account Hijacking

Credential theft via phishing or credential stuffing. Attacker gains full control of your cloud account.

Insecure APIs

Poorly designed cloud interfaces expose services. APIs are the primary attack surface in cloud.

DoS / DDoS

Overwhelming cloud resources with traffic to deny legitimate users access.

Insider Threats

Malicious or negligent employees with cloud access can leak or delete critical data.

Misconfiguration

Open S3 buckets, public databases, wrong IAM policies. Most common cause of breaches.

Shared Tech Vulnerabilities

VM escape, side-channel attacks in multi-tenancy environments. One VM affects others.

APTs

Advanced Persistent Threats — sophisticated, long-term targeted attacks by nation-states or organized hackers.

Information Security — The CIA Triad

Every security decision in cloud is evaluated against the CIA Triad:

🔒 C — Confidentiality

Only authorized users can access data.

  • Encryption at rest (AES-256)
  • Encryption in transit (TLS 1.3)
  • Data masking and tokenization
  • Key Management Services (KMS)
  • Data classification policies
✅ I — Integrity

Data is accurate and hasn't been tampered with.

  • Hashing algorithms (SHA-256)
  • Digital signatures and PKI certificates
  • Audit logs, checksums, versioning
  • Immutable storage (write-once policies)
⚡ A — Availability

Systems and data are accessible when needed.

  • Redundancy and failover mechanisms
  • Auto-scaling and load balancing
  • Disaster Recovery (DR) with RTO/RPO targets
  • Business Continuity Plans (BCP)
RTO vs RPO: RTO (Recovery Time Objective) = maximum downtime allowed. RPO (Recovery Point Objective) = maximum data loss allowed. E.g., RTO=4hrs, RPO=1hr means you must be back up within 4 hours and can only lose up to 1 hour of data.

Identity Management and Access Control

Identity Management = verifying WHO you are. Access Control = deciding WHAT you can do.

Identity Management Tools

  • IAM (Identity and Access Management) — AWS IAM, Azure AD, GCP IAM
  • Multi-Factor Authentication (MFA) — requires two or more proofs of identity (password + OTP)
  • Single Sign-On (SSO) — one login for multiple services, uses SAML / OAuth 2.0 / OIDC protocols
  • Privileged Access Management (PAM) — extra controls for admin/root-level accounts
  • Directory Services — LDAP, Active Directory (central database of users)

IAM in AWS (5 marks answer): AWS IAM lets you create Users (individual identities), Groups (collection of users), Roles (temporary permissions for services), and Policies (JSON documents defining allowed/denied actions). Key principle: Least Privilege — give only the minimum permissions needed. MFA should be enabled for all root and admin accounts.

Cloud Security Design Principles

01 — Defense in Depth

Multiple layered security controls. If one layer fails, others protect you. No single point of failure.

02 — Least Privilege

Grant only the minimum permissions required for the task. Don't give admin access to everyone.

03 — Separation of Duties

No single person has end-to-end control over critical processes. Multiple approvals needed.

04 — Fail Secure

When a system fails, it defaults to a secure state (deny all). Don't accidentally open access on failure.

05 — Zero Trust Architecture

Verify every request regardless of network origin or user. No implicit trust inside the network.

06 — Immutable Infrastructure

Replace components rather than patching running systems. Reduces configuration drift and hidden vulnerabilities.

07 — Shift Left Security

Embed security from the design phase — DevSecOps. Don't add security as an afterthought.

08 — Data Minimization

Collect and retain only data that is strictly necessary. Less data = smaller attack surface.

Cloud Security Management Frameworks

CSA CCM (Cloud Controls Matrix)

Cloud Security Alliance standard. 197 control objectives across 17 domains. Most comprehensive cloud-specific framework.

NIST SP 800-144

US National Institute of Standards & Technology guidelines on security and privacy in public cloud computing.

ISO/IEC 27017

Cloud-specific information security controls. Extends ISO 27001 (general InfoSec) with cloud-specific guidance.

ISO/IEC 27018

Protection of Personally Identifiable Information (PII) in public cloud. Privacy-focused standard.

SOC 2 Type II

Audits cloud providers on Security, Availability, Processing Integrity, Confidentiality, and Privacy over time.

FedRAMP

US Government cloud security authorization framework. Required for cloud providers serving federal agencies.

CIS Benchmarks

Technical security configuration standards for cloud environments (AWS, Azure, GCP). Specific, actionable checklists.

GDPR / HIPAA / PCI-DSS

Regulatory frameworks: GDPR (EU data privacy), HIPAA (healthcare), PCI-DSS (payment cards). All have cloud implications.

Exam Tip — Frameworks (5 marks): Mention CSA CCM (cloud-specific, 17 domains), ISO 27017 (cloud controls), NIST SP 800-144 (US guidelines), and SOC 2 (audit). Note that ISO 27001 is the parent standard for general InfoSec, while 27017 extends it for cloud.

5.2 Host and Data Security in SaaS, PaaS, IaaS

Shared Responsibility Model

The most important concept in cloud security: Security is a shared responsibility between the cloud provider and the customer. How much you manage depends on the service model.

Simple rule: More abstraction (IaaS → PaaS → SaaS) = less security burden on the customer, but also less control.
Layer IaaS (e.g., AWS EC2) PaaS (e.g., App Engine) SaaS (e.g., Gmail, Salesforce)

Host & Data Security in IaaS (e.g., AWS EC2, Azure VMs)

In IaaS, you get virtual machines. You are responsible for the OS and everything above. The provider secures the physical hardware and hypervisor.

Host Security (Customer's Job)
  • VM hardening using CIS Benchmarks & OS patching
  • Host-based Intrusion Detection (HIDS / OSSEC)
  • Hypervisor security and VM isolation guarantees
  • Secure Boot and TPM (Trusted Platform Module)
  • Network segmentation using VPCs, Security Groups, NACLs
Data Security (Customer's Job)
  • Encryption at rest — AWS EBS, Azure Disk Encryption, GCP CSEK
  • Encryption in transit — VPN tunnels, TLS 1.3
  • Backup and snapshot policies with geo-redundancy
  • Data Loss Prevention (DLP) tools
  • Key Management Services (KMS) — customer-managed keys (CMK)

Host & Data Security in PaaS (e.g., Google App Engine, Elastic Beanstalk)

In PaaS, the provider manages the OS, runtime, and middleware. You focus on your application code and data. Security responsibilities shift more to the provider.

Application Security (Customer)
  • Container security — Docker image scanning, Kubernetes RBAC
  • Runtime protection tools (Falco, Sysdig, Aqua)
  • Dependency and library vulnerability scanning (Snyk, Dependabot)
  • Secure CI/CD pipelines — SAST, DAST, SCA in pipeline
  • Secrets management — HashiCorp Vault, AWS Secrets Manager
Data Security
  • Database encryption — TDE (Transparent Data Encryption)
  • Object storage encryption — S3 SSE, Azure SSE, GCS CMEK
  • Secure API design — OAuth, rate limiting, input validation
  • Database access controls, audit logging, and query monitoring
  • Environment variable and config encryption for app settings

Host & Data Security in SaaS (e.g., Salesforce, Microsoft 365)

In SaaS, the provider manages almost everything. Your responsibility is mainly data governance, access control, and user management.

Provider Responsibility
  • Multi-tenant isolation & VM separation
  • Automated OS patch management
  • Physical & network infrastructure security
  • Certifications: SOC 2, ISO 27001, CSA STAR
Customer Responsibility
  • User provisioning & deprovisioning (joiner-mover-leaver)
  • Enforce MFA and conditional access
  • Shadow IT discovery & control
  • Third-party app integration review
  • SaaS Security Posture Management (SSPM)
Data Security Tools
  • CASB — Cloud Access Security Broker (visibility & control over SaaS)
  • DLP — Data Loss Prevention (prevent data exfiltration)
  • Data backup and export / portability policies
  • End-to-end encryption (customer keys)
  • Data residency and sovereignty compliance
CASB is a security policy enforcement point between cloud users and cloud service providers. It gives visibility into what SaaS apps employees use and enforces policies (block risky apps, prevent data leakage). Examples: Microsoft Defender for Cloud Apps, Netskope.

5.3 Cloud Cost Optimization, Budget Management, Reserved vs Spot Instances, FinOps

Why Cloud Cost Optimization?

According to Gartner, 30-35% of cloud spend is wasted — idle resources, oversized instances, forgotten storage, data transfer costs. Cost optimization means getting maximum value from every rupee/dollar spent on cloud.

Key Cost Drivers in Cloud

  • Compute (EC2, VMs, GCE) — biggest cost center
  • Storage (S3, EBS, Blob, GCS)
  • Data Transfer — egress charges (data leaving cloud) add up fast
  • Managed Services (RDS, EKS, Lambda) — convenient but costly
  • Licensing — BYOL (Bring Your Own License) vs provider-managed

Optimization Goals

  • Eliminate waste — right-sizing, removing zombie/idle resources
  • Maximize value — measure cost per workload/business outcome
  • Improve forecasting & financial accountability
  • Enable engineering autonomy with guardrails (policy limits)

On-Demand vs Reserved vs Spot / Preemptible Instances

On-Demand (0% discount)
  • Pay per hour/second, no commitment
  • Most expensive — maximum flexibility
  • No interruption risk
  • Best for: unpredictable, short-term workloads; testing/dev
Reserved Instances (up to 72% off)
  • Commit to 1 or 3 years (upfront or monthly payments)
  • AWS: up to 72% | Azure: 65% | GCP CUDs: 57%
  • Types: Standard (locked) vs Convertible (flexible)
  • Best for: steady-state, predictable production workloads
  • AWS also offers Savings Plans (more flexible)
Spot / Preemptible (60-90% off)
  • Use spare/unused cloud capacity at deep discount
  • Can be interrupted — 2-minute warning on AWS/Azure
  • GCP Preemptible: max 24h, 30-second notice
  • Best for: batch jobs, ML training, CI/CD pipelines (fault-tolerant)
  • AWS Spot Fleet / Azure Spot VMs / GCP Spot VMs
Simple Rule: On-Demand = flexible but expensive. Reserved = commit & save big for predictable work. Spot/Preemptible = huge savings but can be taken away anytime — only use for interruptible workloads.

Cloud Budget Management

Budget management is about setting spending limits, tracking costs, and ensuring accountability across teams.

Cloud Provider Tools

  • AWS: AWS Budgets + Cost Explorer
  • Azure: Cost Management + Billing
  • GCP: Budgets & Alerts + Cost Dashboard

Budget Strategies

  • Chargeback & Showback — Allocate cloud costs to specific teams/projects. Chargeback = team actually pays; Showback = team sees costs but doesn't pay
  • Cost Tagging — Tag all resources with env (prod/dev/test), team, project, app. Essential for tracking who is spending what
  • Budget Policies — Enforce spending limits per department or business unit. Auto-stop resources when limit reached
  • Cost Anomaly Detection — AI-driven alerts for unexpected spending spikes (AWS Cost Anomaly Detection)

Best Practices

  • Regular cost reviews — weekly team, monthly executive
  • Cloud cost governance policy and FinOps charter
  • Executive dashboards for spend visibility and forecasting
  • Dedicated FinOps team or Cloud Center of Excellence (CCoE)

FinOps — Financial Operations for Cloud

FinOps is a cultural and operational practice that brings together Finance, Engineering, and Business teams to maximize the business value of cloud spending. It's not just a tool — it's a mindset and organizational practice.

6 Core FinOps Principles (FinOps Foundation)

1. Teams Collaborate

Finance + Engineering + Product work together. Breaking silos is essential for cost accountability.

2. Value-Driven Decisions

Focus on business value, not just cost reduction. Spending more on high-value services may be right.

3. Everyone Owns Cost

Decentralized accountability across all teams. Engineers understand and own the cost of what they build.

4. Timely Data Access

Real-time cost visibility for fast decisions. Stale data = poor decisions.

5. Centralized FinOps Team

CCoE / FinOps team drives best practices, standards, and tooling across the organization.

6. Variable Cost Advantage

Leverage cloud's elasticity as a competitive asset — scale up when needed, scale down to save.

FinOps Lifecycle: Inform → Optimize → Operate

📊 INFORM (Visibility & Allocation)

Understand where money is being spent. Create cost allocation with tags. Build dashboards for stakeholders.

Tools: AWS Cost Explorer, Azure Cost Management, GCP Billing Console, CloudHealth, Apptio Cloudability

⚙️ OPTIMIZE (Reduce Waste & Right-Size)

Identify and eliminate idle resources. Purchase reservations and savings plans. Right-size overprovisioned instances.

Tools: AWS Compute Optimizer, Azure Advisor, GCP Recommender, Spot.io, ParkMyCloud, Densify

🔄 OPERATE (Continuous Improvement)

Establish governance policies and automation. Build culture of cost accountability. Automate enforcement of tagging and policies.

Tools: Terraform (IaC), AWS Organizations SCPs, Azure Policy, GCP Org Policies, Infracost

Key Takeaway for Exam: FinOps is NOT just about saving money. It's about making sure every dollar spent on cloud generates maximum business value. The lifecycle: first understand your spending (Inform), then eliminate waste (Optimize), then continuously monitor and govern (Operate).
